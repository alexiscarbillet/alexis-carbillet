Chapter 1 Home/scratch mechanism

The temporary folder contains a script and some data processed in the script. The script is a copy of one which is on GitLab. The data in this folder are copies from the original ones. Inside this command system(), there is a command line to copy/paste the data into a temporary folder called temporary. In fact, it can contains any linux command, such as “rm” or “mkdir”. The temporary folder is in the datastore in order to avoid having to import a large amount of data into GitLab and thus have an important response time. The R script contained in the temporary folder will not be modified. The changes will be made in the script in GitLab. The tests will be performed from a sample of data also in GitLab. If the changes match the expectations, then the new version will be put in the temporary folder. This temporary folder is deleted at each end of use.

Here are the command lines of system() command that I used in a script serving as template to the project which was proposed to me:

At the beginning of the script, in order to access the databases:

system("cp ~/datastore/AlfaDWbh6/ext-rdy/sampro/htmdynfru/watercolors.json ~/datastore/AlfaDWbh6/ext-rdy/temporary") 
# temporary copy of watercolors.json
system("cp ~/datastore/AlfaDWbh6/ext-rdy/sampro/htmdynfru/thumbs.json ~/datastore/AlfaDWbh6/ext-rdy/temporary") 
# temporary copy of thumbs.json

At the end of the script, to delete the temporary databases:

system("rm ~/datastore/AlfaDWbh6/ext-rdy/temporary/watercolors.json ") 
# delete the temporary copy of watercolors.json
system("rm ~/datastore/AlfaDWbh6/ext-rdy/temporary/thumbs.json") 
# delete the temporary copy of thumbs.json

In these examples, 2 files are copied from the datastore to a temporary file. Then they are used by the code which is also in this temporary file. At the end of the code, they are deleted. It allows the user to isolated each file and to modify them locally if necessary without changing the original one. It is important if many people are using them for differents codes, it avoid contradictions between the different commands and also avoid to work with data already processed by someone else without knowing it.

This concludes the home/scratch mechanism section. Next, the generation of a dynamic report will be studied.
